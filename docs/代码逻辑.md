# 代码逻辑

## 1. 文档说明

本文档重点说明项目代码的“执行逻辑”，也就是系统在运行时每一步是如何串起来的。  
如果说《代码结构说明》主要回答“有哪些文件、每个文件做什么”，那么本文档主要回答“请求来了以后，代码是怎么跑起来的”。

本文按照以下主线展开：

1. 系统启动逻辑
2. 文档上传与入库逻辑
3. 问答检索逻辑
4. 特殊增强逻辑
5. 前端交互逻辑

## 2. 系统启动逻辑

系统启动入口在 [backend/app.py](/Users/bigfish/Desktop/industrial-rag/backend/app.py)。

启动时的执行流程如下：

1. 创建 Flask 应用
2. 从 [backend/config.py](/Users/bigfish/Desktop/industrial-rag/backend/config.py) 读取配置
3. 初始化数据库 `db.init_app(app)`
4. 创建上传目录、向量索引目录、聊天图片目录
5. 调用 `db.create_all()` 建表
6. 调用 `ensure_document_schema()` 检查数据库字段是否完整
7. 加载 FAISS 索引
8. 如果索引文件不存在，或者索引中的 chunk 数与数据库不一致，就调用 `rebuild_index()` 重建索引

这一步完成后，系统就进入“可上传、可问答”的运行状态。

## 3. 文档上传与入库逻辑

文档上传接口是：

- `/api/upload`

对应函数在 [backend/app.py](/Users/bigfish/Desktop/industrial-rag/backend/app.py) 的 `upload_file()`。

### 3.1 上传前检查

`upload_file()` 接到请求后，首先做基础校验：

1. 是否真的上传了文件
2. 文件名是否为空
3. 扩展名是否在允许范围内
4. 分类值是否合法

同时还会调用：

- `sanitize_upload_filename()`
  清洗文件名，避免路径穿越和危险字符

- `build_unique_filepath()`
  如果目录里已经有同名文件，就自动给文件名加后缀，避免覆盖

### 3.2 文件保存与数据库建档

校验通过后，后端先把原始文件保存到上传目录，然后在 `Document` 表中创建一条记录，状态先写成：

- `processing`

这样做的目的是：即使后续解析流程较长，数据库里也已经有这份文档的元信息。

### 3.3 文本提取逻辑

接下来进入解析阶段，逻辑分三种：

#### 情况一：图片文件

如果是 `.png/.jpg/.jpeg/.bmp/.webp` 这类图片，则调用：

- `summarize_image()`

由视觉模型直接生成一段可检索文本。

#### 情况二：普通文档

如果是 `.docx/.txt`，或者 PDF 中能直接提取文字，则调用：

- `parse_file()`

由 [backend/doc_parser.py](/Users/bigfish/Desktop/industrial-rag/backend/doc_parser.py) 把正文转成文本。

#### 情况三：扫描版 PDF

如果 PDF 文字提取结果太少，则认为它更像图片型 PDF。

此时会执行：

1. `render_pdf_pages()`
   把 PDF 页面渲染为图片
2. `extract_pdf_page_text()`
   对页面做 OCR 风格文字抽取
3. 如果 OCR 风格抽取失败，再退回 `summarize_image()`

这一步的目的是尽可能把图片型 PDF 转成可检索文本。

### 3.4 文本分块与向量化

拿到完整文本后，进入 RAG 入库阶段：

1. 调用 `split_text()`
   把长文本切成多个 chunk
2. 调用 `get_embeddings_batch()`
   批量生成 chunk 向量
3. 将 chunk 文本和向量写入 `Chunk` 表
4. 更新 `Document.chunk_count`
5. 将向量增量写入 FAISS 索引 `add_embeddings()`

到这一步，文档就正式进入知识库，可以参与后续问答。

### 3.5 上传失败兜底

如果入库中途发生异常，`upload_file()` 会做清理：

1. 数据库回滚
2. 删除已保存的原始文件
3. 删除已创建的 `Document` 记录
4. 删除 PDF 解析过程中的临时图片目录

这样可以避免知识库里出现“半成功”的脏数据。

## 4. 问答逻辑

问答接口是：

- `/api/chat`

对应函数在 [backend/app.py](/Users/bigfish/Desktop/industrial-rag/backend/app.py) 的 `chat()`。

### 4.1 接收问题

前端发起提问时，后端会接收三类信息：

1. `question`
   用户问题文本

2. `image`
   可选，用户在问答时附带的图片

3. `doc_id`
   可选，表示本次只在某一份文档内检索

### 4.2 空知识库处理

如果数据库里没有任何 chunk：

- 没附图时，返回“知识库为空，请先上传文档”
- 附图时，直接走 `ask_multimodal_llm()`，让模型只根据图片回答

### 4.3 检索范围判断

如果用户没有选定某个文件：

- 调用 `search_similar()`
- 在整个 FAISS 索引中做全知识库检索

如果用户选择了某个文件：

- 调用 `search_within_document()`
- 只在该文档的 chunk 内部做相似度计算

这就是“全库检索”和“单文件检索”切换的核心逻辑。

### 4.4 检索结果增强

初始检索得到的只是若干相似 chunk。  
为了让最终答案更稳定，系统在后端增加了多种“上下文增强策略”。

#### 1. 结果分散

调用：

- `select_diverse_results()`

作用是防止一个文档的多个 chunk 把结果全部占满。

#### 2. 表格类问题增强

当问题明显包含这些倾向时：

- 方法
- 作用
- 参数
- 元素
- 对照
- 类别

系统会判定为表格类问题，并执行：

- `augment_table_neighbors()`

作用是把表格相邻 chunk 一起补进上下文，尽量把整张表带全。

#### 3. 连续段落问题增强

当问题更像“过程、步骤、原理、分类、基本过程”时，系统会执行：

- `augment_context_neighbors()`

作用是补前后相邻 chunk，避免正文内容因为切块而断头断尾。

#### 4. 特定问题的局部段落抽取

针对“化学热处理由哪些基本过程组成”这类问题，系统又额外做了：

- `build_focused_continuity_context()`

作用是从连续 chunk 中提取一段更干净的正文，避免把表格尾巴和下一页 OCR 噪声一起送给模型。

### 4.5 规则化答案生成

对于“基本过程组成”这类题目，系统不是完全依赖大模型总结，而是额外做了一层规则抽取：

- `build_process_answer_from_context()`

逻辑是：

1. 在聚焦后的正文里找到“化学热处理通常由四个基本过程组成”
2. 按固定标题定位四条内容：
   - 介质中的化学反应
   - 渗剂扩散
   - 相界面反应
   - 被吸附并溶入的渗入元素向工件内部扩散
3. 去掉分块 overlap 造成的重复编号污染
4. 最后拼成稳定输出

这一步的意义是：

即使大模型有时会漏项，系统也能对特定高频题给出更稳定的正确答案。

### 4.6 调用大模型生成最终回答

如果没有命中规则化抽取，就进入普通回答流程：

- 纯文本问题：`ask_llm()`
- 图文联合问题：`ask_multimodal_llm()`

在这一步里，后端已经把上下文整理好了，模型只需要根据上下文回答。

### 4.7 保存历史记录

回答生成后，系统会把以下内容写入 `ChatHistory`：

- 问题
- 回答
- 来源摘要

因此前端的“最近问答”和“历史问答详情弹层”才能正常工作。

## 5. 前端交互逻辑

前端逻辑主要在 [frontend/app.js](/Users/bigfish/Desktop/industrial-rag/frontend/app.js)。

### 5.1 页面初始化

页面加载后会自动执行：

- `loadDocuments()`
- `loadHistory()`

也就是：

1. 拉取知识库文档列表
2. 拉取最近问答历史

### 5.2 上传逻辑

用户上传文件时：

1. 前端检查扩展名是否合法
2. 组装 `FormData`
3. 调用 `/api/upload`
4. 根据返回结果刷新知识库列表和统计信息

### 5.3 文档展示逻辑

`loadDocuments()` 返回文档列表后，前端会做三件事：

1. `updateStats(docs)`
   更新知识库统计卡片

2. `renderDocuments(docs)`
   按分类分组展示文档列表

3. `updateQueryDocOptions(docs)`
   更新“本次检索范围”下拉框中的文件选项

### 5.4 单文件检索范围选择

前端问答区现在支持：

- 全知识库检索
- 指定单个文件检索

相关逻辑包括：

- `queryDocSelect`
  用户选择本次问答检索范围

- `updateQueryScopeIndicator()`
  在界面上显示“当前仅检索：xxx.pdf”

发送问题时，前端会把 `doc_id` 一起传给后端。

### 5.5 问答发送逻辑

`sendQuestion()` 的执行流程如下：

1. 获取问题文本
2. 获取可选附图
3. 获取可选 `doc_id`
4. 显示用户消息
5. 显示机器人 loading 状态
6. 调用 `/api/chat`
7. 返回后显示模型回答
8. 刷新历史记录

### 5.6 历史问答查看逻辑

最近问答支持点击查看完整详情。

实现逻辑：

- `loadHistory()`
  拉取历史记录并渲染卡片

- `openHistoryModal()`
  打开弹层并显示完整问题与回答

- `closeHistoryModal()`
  关闭弹层

## 7. 总结

从执行逻辑角度来看，这个系统不是简单地“上传文件后问一句模型”，而是实现了完整的链路：

1. 文件入库
2. 文档解析
3. 文本分块
4. 向量化
5. 向量索引
6. 检索增强
7. 模型生成
8. 历史保存
9. 前端展示

这也是它能够作为毕业设计系统进行展示的核心原因：  
不是只调用大模型，而是围绕工业文档知识问答做了一套完整的工程实现。
