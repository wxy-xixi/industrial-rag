# 代码结构说明

## 1. 项目整体概述

本项目是一个面向工业知识问答场景的多模态 RAG 系统，整体由前端展示层、后端接口层、文档解析层、向量检索层和模型生成层组成。

系统的核心流程是：

1. 用户上传文档或图片
2. 后端解析内容并切分为文本块
3. 文本块经过向量化后写入数据库与 FAISS 索引
4. 用户提问时，系统先做向量检索，再拼接上下文
5. 最后调用大模型生成回答，并返回参考来源

项目目录结构如下：

```text
industrial-rag/
├── backend/                 # Flask 后端
│   ├── app.py               # 后端主入口与业务编排
│   ├── config.py            # 配置管理
│   ├── models.py            # 数据库模型
│   ├── doc_parser.py        # 文档解析与文本分块
│   ├── rag_engine.py        # 向量检索与模型调用
│   ├── uploads/             # 上传文件目录
│   ├── vector_store/        # FAISS 索引目录
│   └── industrial_rag.db    # 本地 SQLite 数据库
├── frontend/                # 前端页面
│   ├── index.html           # 页面结构
│   ├── app.js               # 前端交互逻辑
│   └── style.css            # 页面样式
├── docs/                    # 项目文档
└── README.md                # 项目说明
```

## 2. 后端结构说明

### 2.1 `backend/app.py`

`app.py` 是整个后端系统的主入口，负责业务流程的组织与调度，可以理解为系统的“控制中心”。

主要职责包括：

- 创建 Flask 应用并加载配置
- 初始化数据库和 FAISS 索引
- 检查数据库表结构是否完整
- 提供前端静态页面访问
- 提供上传、问答、文档管理、历史记录等接口

核心接口如下：

- `/api/upload`
  作用：上传文档或图片，解析文本，切分为 chunk，向量化并写入数据库和 FAISS

- `/api/chat`
  作用：接收用户问题，完成检索、上下文增强和大模型回答

- `/api/documents`
  作用：返回当前知识库文档列表

- `/api/documents/<id>`
  作用：删除指定文档

- `/api/documents/<id>/category`
  作用：修改文档分类

- `/api/history`
  作用：返回历史问答记录

`app.py` 中还包含一些关键的辅助函数：

- `normalize_category`
  作用：规范文档分类输入，避免写入非法值

- `sanitize_upload_filename`
  作用：清洗上传文件名，避免路径穿越风险

- `build_unique_filepath`
  作用：防止同名文件覆盖

- `select_diverse_results`
  作用：从全库候选中做结果分散，避免某一个文档垄断 top-k

- `search_within_document`
  作用：支持用户选择“只在某个文件内检索”

- `augment_table_neighbors`
  作用：对于表格类问题，补充表格相邻 chunk，尽量保留整表上下文

- `augment_context_neighbors`
  作用：对于流程、步骤、分类类问题，补充前后相邻 chunk

- `build_focused_continuity_context`
  作用：针对“基本过程/步骤”这类问题，从连续 chunk 中抽取更干净的局部段落

### 2.2 `backend/config.py`

`config.py` 是系统的配置中心，用来统一管理运行参数。

主要职责：

- 自动读取项目根目录下的 `.env`
- 设置数据库连接地址
- 设置上传目录、索引目录、临时图片目录
- 设置模型名称、检索参数和服务端口

常见配置项包括：

- `SQLALCHEMY_DATABASE_URI`
- `UPLOAD_FOLDER`
- `VECTOR_STORE_DIR`
- `CHAT_IMAGE_FOLDER`
- `TEXT_MODEL`
- `VL_MODEL`
- `OCR_MODEL`
- `TOP_K`
- `RETRIEVAL_CANDIDATES`
- `MAX_CHUNKS_PER_DOC`

这样做的好处是：开发、演示、部署环境可以灵活切换，而不需要修改业务代码。

### 2.3 `backend/models.py`

`models.py` 定义了数据库中的三张核心表：

#### `Document`

保存知识库文件信息，包括：

- 文件名
- 文件类型
- 文件大小
- 分块数量
- 文档分类
- 上传时间
- 处理状态

#### `Chunk`

保存 RAG 检索使用的文本块信息，包括：

- 所属文档 `doc_id`
- 分块内容 `content`
- 向量数据 `embedding`
- 分块序号 `chunk_index`

#### `ChatHistory`

保存每次问答记录，包括：

- 用户问题
- 模型回答
- 参考来源
- 创建时间

其中，`Document` 和 `Chunk` 是一对多关系：一个文档会被拆成多个文本块。

### 2.4 `backend/doc_parser.py`

`doc_parser.py` 负责文档内容提取与预处理，是“解析层”。

主要函数如下：

- `parse_file`
  按文件后缀分发不同的解析方法

- `parse_pdf`
  提取普通 PDF 中的文字内容

- `parse_docx`
  提取 Word 文档中的正文文本

- `parse_txt`
  读取文本文件内容

- `render_pdf_pages`
  将扫描版 PDF 渲染为图片，用于 OCR 风格解析

- `split_text`
  将长文本切分成多个 chunk，供向量检索使用

这一层的目标是把不同类型的文件尽可能统一转换成“可检索的文本表示”。

### 2.5 `backend/rag_engine.py`

`rag_engine.py` 是系统的智能核心，负责向量处理、相似检索和模型调用。

可以分为四部分：

#### 1. 向量与索引管理

- `load_index`
- `rebuild_index`
- `add_embeddings`

这些函数负责 FAISS 索引的加载、重建和增量更新。

#### 2. 文本向量能力

- `get_embedding`
- `get_embeddings_batch`

作用是把文本转换成向量，用于后续相似度检索。

#### 3. 相似检索能力

- `search_similar`
  在 FAISS 中检索与问题最相似的 chunk

- `looks_like_table_chunk`
  判断某个 chunk 是否更像表格/字段对照内容

#### 4. 模型调用能力

- `ask_llm`
  文本问答，输入问题和上下文，输出最终回答

- `summarize_image`
  对图片生成可检索摘要，供图片入库使用

- `extract_pdf_page_text`
  对扫描版 PDF 页面执行 OCR 风格抽取

- `ask_multimodal_llm`
  在问答时结合图片和文本上下文联合生成答案

这一层的作用就是：把检索和模型能力封装起来，为 `app.py` 提供可调用的智能模块。

## 3. 前端结构说明

### 3.1 `frontend/index.html`

`index.html` 定义了页面的整体结构，主要可以分为左右两大区域：

#### 左侧区域

- 系统 Logo 和简介
- 文档上传区
- 知识库统计区
- 已上传文档列表
- 最近问答记录

#### 右侧区域

- 系统标题与展示卡片
- 聊天消息区
- 问答输入区
- 历史问答详情弹层

前端当前还支持：

- 上传时选择分类
- 问答时选择“全知识库 / 单文件检索”
- 显示当前检索范围提示

### 3.2 `frontend/app.js`

`app.js` 是前端行为逻辑的核心。

主要功能包括：

- 文件上传
- 文档列表加载
- 分类筛选与文档搜索
- 修改文档分类
- 删除文档
- 发送文本/图文问答请求
- 展示问答消息
- 查看历史问答详情
- 控制“本次检索范围”
- 高亮显示“当前仅检索某文件”

可以理解为：`app.js` 负责把用户操作转成 API 请求，再把后端返回的数据渲染到页面上。

### 3.3 `frontend/style.css`

`style.css` 负责整套界面的视觉呈现。

主要实现内容：

- 深色工业科技风布局
- 左右双栏结构
- 左右独立滚动
- 卡片式组件风格
- 聊天气泡样式
- 分类标签与状态徽标
- 历史详情弹层
- 当前检索范围提示样式

这一层不涉及核心算法，但直接决定系统展示效果和答辩演示体验。

## 4. 系统运行流程

### 4.1 文档入库流程

1. 用户在前端上传文档或图片
2. 前端调用 `/api/upload`
3. 后端保存文件并写入 `Document`
4. 后端解析文件文本
5. 文本分块
6. 调用 embedding 接口批量向量化
7. 将 chunk 写入数据库
8. 将向量写入 FAISS 索引

### 4.2 问答流程

1. 用户输入问题
2. 前端可选择检索范围：全知识库或某个文件
3. 前端调用 `/api/chat`
4. 后端执行向量检索
5. 根据问题类型进行上下文增强
   - 表格问题：补表格相邻块
   - 连续性问题：补前后 chunk
   - 特定流程类问题：抽取局部干净段落
6. 后端调用大模型生成回答
7. 保存历史记录
8. 前端展示答案与来源

## 5. 当前代码中的几个关键设计点

### 5.1 RAG 检索不是简单关键词搜索

系统不是直接做字符串匹配，而是先将 chunk 和问题向量化，再基于 FAISS 做语义相似检索，因此能支持更自然语言化的问题。

### 5.2 支持扫描版 PDF

普通 PDF 提取不到文字时，会自动走页面渲染与 OCR 风格解析流程，因此可以处理图片型 PDF。

### 5.3 支持多模态问答

系统不只支持文本入库，也支持：

- 图片入库生成摘要
- 提问时附加图片
- 图文联合问答

### 5.4 支持全库检索与单文件检索切换

用户可以自由决定：

- 本次问题在整个知识库中检索
- 还是只在某一份文档中检索

这个功能在答辩展示时很有价值，因为可以直观说明系统具备“全局问答”和“局部精确问答”两种模式。

## 6. 答辩时可以怎样讲代码结构

建议按照“五层结构”来介绍：

1. 展示层
   前端页面、交互逻辑和样式展示

2. 接口层
   Flask 提供上传、问答、历史记录和文档管理接口

3. 解析层
   各类文件统一转换成可检索文本

4. 检索层
   chunk、embedding、FAISS、上下文增强策略

5. 生成层
   将检索到的资料交给大模型生成最终回答

可以用一句话总结为：

“本系统通过文档解析、文本分块、向量检索和大模型生成，构建了一个支持工业文档与图片知识利用的多模态 RAG 智能问答平台。”

## 7. 总结

从代码层面看，这个项目并不是一个简单的“网页 + 调模型”，而是围绕工业知识问答完整实现了：

- 知识入库
- 文档解析
- 向量检索
- 上下文增强
- 大模型回答
- 历史记录管理
- 前端交互展示

整体结构已经具备一个完整 RAG 原型系统应有的核心模块，既适合课程设计和毕业设计展示，也便于后续继续扩展更强的 OCR、表格解析、重排序和部署能力。
