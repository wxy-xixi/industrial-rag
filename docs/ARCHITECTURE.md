# 系统架构与流程说明

## 一、系统目标

本系统面向工业知识问答场景，目标是构建一个支持文本资料与工业图片的多模态 RAG 平台，实现“知识入库、语义检索、生成回答”的完整闭环。

## 二、总体架构

系统整体可分为五层：

1. 表现层：前端页面，负责文件上传、问答交互、结果展示。
2. 接口层：Flask API，负责请求分发、参数校验和流程编排。
3. 知识处理层：文档解析、图片摘要、文本分块、向量化。
4. 检索层：FAISS 向量索引与数据库回查。
5. 模型层：DashScope 提供 embedding、文本生成和视觉理解能力。

## 三、模块说明

### 1. 前端模块

前端位于 `frontend/`，主要包含：

- `index.html`：页面结构。
- `style.css`：系统样式。
- `app.js`：上传、问答、历史加载、统计展示等逻辑。

前端功能包括：

- 文档/图片上传。
- 文本问答。
- 图文联合问答。
- 文档列表显示。
- 知识库统计。
- 最近问答历史展示。

### 2. 后端接口模块

后端位于 `backend/app.py`，核心接口包括：

- `/api/upload`：上传文档或图片并完成入库。
- `/api/chat`：执行文本问答或图文联合问答。
- `/api/documents`：返回知识库文档列表。
- `/api/documents/<id>`：删除指定文档。
- `/api/history`：获取最近问答历史。

### 3. 文档处理模块

文档解析位于 `backend/doc_parser.py`，支持：

- PDF 文本提取。
- DOCX 文本提取。
- TXT 文本读取。
- 文本分块。

图片文件不走传统文档解析，而是先由视觉模型生成描述文本，再进入统一的文本分块和向量化流程。

### 4. 数据模型模块

数据表定义位于 `backend/models.py`：

- `Document`：保存文件名、类型、大小、分块数、上传时间、状态。
- `Chunk`：保存每个文本分块及其向量。
- `ChatHistory`：保存问题、回答和来源信息。

### 5. RAG 引擎模块

核心检索与生成逻辑位于 `backend/rag_engine.py`，包括：

- 文本 embedding 获取。
- 批量向量化。
- FAISS 索引加载、保存、重建。
- 相似分块检索。
- 文本生成问答。
- 图片摘要生成。
- 图文联合问答。

## 四、核心数据流

### 1. 文档入库数据流

```text
用户上传文件
    ->
后端校验与安全保存
    ->
文本解析 或 图片摘要生成
    ->
文本分块
    ->
embedding 向量化
    ->
写入数据库 Chunk 表
    ->
写入 FAISS 索引
```

### 2. 问答数据流

```text
用户输入问题
    ->
问题向量化
    ->
FAISS 检索相关分块
    ->
数据库回查具体内容
    ->
拼接上下文
    ->
大模型生成回答
    ->
返回答案与来源
```

### 3. 图文联合问答数据流

```text
用户输入问题 + 附加图片
    ->
文本问题向量化
    ->
FAISS 检索知识库上下文
    ->
图片 + 检索上下文 + 用户问题
    ->
多模态模型生成回答
```

## 五、为什么使用 RAG

传统大模型直接回答工业问题存在三个问题：

1. 容易脱离企业文档产生幻觉。
2. 不具备企业私有知识。
3. 回答缺乏可追溯性。

RAG 的优势在于：

1. 先检索再生成，回答更贴近真实资料。
2. 参考来源可展示，可解释性更强。
3. 便于知识库持续扩展。

## 六、为什么使用 FAISS

如果直接对数据库中所有分块逐条计算相似度，随着数据量增大，性能会快速下降。

引入 FAISS 的原因是：

1. 向量检索速度更快。
2. 适合本地原型和中小规模数据。
3. 便于演示“工业知识库系统”的真实检索结构。

## 七、多模态能力设计

系统中的多模态能力分为两类：

### 1. 图片入库

当用户上传工业图片时，系统先调用视觉模型生成一段语义描述，例如：

- 设备名称
- 操作界面文字
- 参数信息
- 告警状态
- 可检索关键词

然后再将这段描述向量化并写入知识库，实现“图片知识化”。

### 2. 图文联合问答

当用户提问时附带一张图片，系统会把：

- 用户问题
- 知识库检索结果
- 附加图片

一起输入多模态模型，生成更完整的答案。

## 八、工程化改进点

为了让系统更适合实际演示和部署，项目还做了以下工程优化：

1. 使用 `.env` 管理配置，避免明文写死密钥。
2. 文件名清洗与唯一路径生成，降低路径穿越和覆盖风险。
3. 上传失败时自动清理文件和数据库记录。
4. 使用安全 DOM 构建，避免前端 XSS。
5. 启动时自动检查并重建 FAISS 索引，保证数据库与索引一致。
