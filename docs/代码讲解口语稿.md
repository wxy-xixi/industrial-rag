# 代码讲解口语稿

## 1. 开场介绍

老师好，下面我从代码实现的角度介绍一下这个系统。

这个项目整体是一个基于 RAG 的工业知识智能问答系统，主要分为前端展示层、后端接口层、文档解析层、向量检索层和模型生成层。

我的核心思路是：

1. 先把工业文档和图片转成可检索文本
2. 再做分块和向量化
3. 用户提问时通过向量检索找到相关内容
4. 最后把检索结果交给大模型生成回答

## 2. 先讲后端主入口

后端主入口在 [backend/app.py](/Users/bigfish/Desktop/industrial-rag/backend/app.py)。

这个文件可以理解为整个系统的“业务总控”。  
它主要负责三类事情：

第一类是系统初始化：

- 创建 Flask 应用
- 加载配置
- 初始化数据库
- 初始化上传目录和 FAISS 索引目录
- 启动时检查数据库表结构和向量索引是否一致

第二类是接口管理：

- 文档上传接口 `/api/upload`
- 问答接口 `/api/chat`
- 文档列表接口 `/api/documents`
- 历史记录接口 `/api/history`

第三类是把检索、上下文增强和模型回答串起来。

也就是说，`app.py` 本身不负责底层算法实现，而是负责把各个模块编排起来。

## 3. 配置和数据库结构

配置文件在 [backend/config.py](/Users/bigfish/Desktop/industrial-rag/backend/config.py)。

这个文件的作用是统一管理项目里的关键参数，比如：

- 数据库地址
- 上传目录
- 向量索引目录
- 模型名称
- 检索条数
- 端口号

这样做的好处是，开发环境和部署环境都可以通过 `.env` 切换，而不需要改代码。

数据库模型在 [backend/models.py](/Users/bigfish/Desktop/industrial-rag/backend/models.py)。

系统里主要有三张表：

1. `Document`
保存每份文档的基本信息，比如文件名、分类、大小、状态

2. `Chunk`
保存文档分块内容和对应向量，这是 RAG 检索的核心数据

3. `ChatHistory`
保存问答历史，用于页面上的最近问答展示和历史详情查看

## 4. 文档入库逻辑

文档入库的主要逻辑在 [backend/app.py](/Users/bigfish/Desktop/industrial-rag/backend/app.py) 和 [backend/doc_parser.py](/Users/bigfish/Desktop/industrial-rag/backend/doc_parser.py)。

上传一份文件以后，大致流程是这样的：

1. 前端把文件发到 `/api/upload`
2. 后端先做文件名清洗和格式校验
3. 把原始文件保存到本地目录
4. 在 `Document` 表里创建一条记录
5. 调用解析模块提取文本
6. 调用 `split_text()` 把长文本切成多个 chunk
7. 调用 embedding 接口把 chunk 转成向量
8. 把 chunk 和向量写入数据库
9. 把向量同步写入 FAISS

这样文档就正式进入知识库了。

## 5. 文档解析逻辑

文档解析模块在 [backend/doc_parser.py](/Users/bigfish/Desktop/industrial-rag/backend/doc_parser.py)。

这里主要是按文件类型分流：

- 普通 PDF：直接抽文字
- DOCX：提取段落文本
- TXT：直接读取
- 扫描版 PDF：先渲染成图片，再走 OCR 风格抽取

这个模块的目标是：把各种格式尽量都统一转换成“可检索文本”。

因为 RAG 的核心前提就是，知识库里的内容必须先变成文本块，才能向量化和检索。

## 6. RAG 核心引擎

RAG 核心逻辑在 [backend/rag_engine.py](/Users/bigfish/Desktop/industrial-rag/backend/rag_engine.py)。

这个文件主要包含四类能力：

第一类：向量能力

- `get_embedding`
- `get_embeddings_batch`

把问题或文本块转换成向量。

第二类：索引能力

- `load_index`
- `rebuild_index`
- `add_embeddings`

负责加载、重建和维护 FAISS 索引。

第三类：检索能力

- `search_similar`

负责在向量空间里找到和问题最接近的 chunk。

第四类：模型调用能力

- `ask_llm`
- `ask_multimodal_llm`
- `summarize_image`
- `extract_pdf_page_text`

这些函数负责调用通义千问完成文本问答、图文问答、图片摘要和扫描页提取。

也就是说，`rag_engine.py` 是系统真正的“智能核心”。

## 7. 问答流程

问答流程主要发生在 [backend/app.py](/Users/bigfish/Desktop/industrial-rag/backend/app.py) 的 `/api/chat`。

执行过程可以分成几步：

1. 接收用户问题
2. 判断用户是否附图
3. 判断用户是否限制在某个文档内检索
4. 做相似度检索
5. 对检索结果做上下文增强
6. 拼接上下文
7. 调用大模型生成回答
8. 保存历史记录
9. 返回答案和参考来源

这个项目里我对问答逻辑做了比较多增强，不是简单 top-k 检索之后就直接丢给模型。

例如：

- 对表格类问题，会优先补表格相关 chunk
- 对流程类问题，会补充前后相邻 chunk
- 对“基本过程组成”这种特定问题，还做了规则化抽取，避免模型漏项

这也是系统回答效果比“直接问模型”更稳定的原因。

## 9. 项目代码里的亮点

如果从代码实现角度总结，我认为这个项目有几个比较值得讲的点：

第一，做了完整的 RAG 工程链路  
不是只调模型，而是把上传、解析、分块、向量化、检索、回答和历史记录完整串起来了。

第二，支持多模态  
除了文本，还支持图片入库和图文联合问答。

第三，支持扫描版 PDF  
当 PDF 无法直接提取文字时，会走页面渲染与 OCR 风格抽取。

第四，支持全库检索和单文件检索切换  
这样既能做开放问答，也能做定向问答。

第五，针对具体问题做了上下文增强  
比如表格问题、流程问题、连续段落问题，都不是机械地用同一套检索策略。
