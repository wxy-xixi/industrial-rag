# 毕设答辩讲解稿

## 1. 项目介绍

各位老师好，我的毕业设计题目是“工业领域多模态 RAG 智能问答系统”。

这个系统面向工业知识查询场景，核心目标是解决两个问题：

1. 工业文档内容分散，人工检索效率低。
2. 传统大模型容易脱离企业资料产生幻觉，回答不够可控。

因此，我设计了一个基于 RAG，也就是检索增强生成的问答平台。系统支持上传工业文档和工业图片，先将知识内容结构化，再结合向量检索和大模型生成，给出更贴近知识库内容的回答。

## 2. 研究思路

我的整体思路可以概括为四步：

1. 采集知识源，包括 PDF、Word、TXT 文档和工业图片。
2. 将原始内容解析成文本，图片通过视觉模型转成语义摘要。
3. 对文本分块后做向量化，并存入 FAISS 索引。
4. 用户提问时先检索，再把检索结果和问题一起交给大模型生成答案。

这样做的优势是：

- 检索阶段保证答案“有依据”。
- 生成阶段保证输出语言自然、便于交互。
- 图片也可以进入知识库，体现多模态能力。

## 3. 系统功能

当前系统主要实现了以下功能：

1. 文档上传，支持 PDF、DOCX、TXT。
2. 图片上传，支持 PNG、JPG、JPEG、BMP、WEBP。
3. 文档解析与文本分块。
4. 向量化与 FAISS 本地索引构建。
5. 基于知识库的智能问答。
6. 提问时附加图片，实现图文联合问答。
7. 文档列表、问答历史和知识库统计展示。

## 4. 系统架构

系统采用前后端分离但由 Flask 统一托管前端页面的轻量结构：

1. 前端负责文件上传、问答交互、结果展示。
2. Flask 后端负责接口编排、文档处理、检索和模型调用。
3. 数据库存储文档元信息、文本分块和问答历史。
4. FAISS 负责本地向量索引。
5. DashScope 提供文本向量模型、文本生成模型和视觉模型。

## 5. 核心流程

### 文档入库流程

1. 用户上传文件。
2. 后端校验格式并安全保存。
3. 如果是文本文件，直接解析内容。
4. 如果是图片，先调用视觉模型生成图片语义摘要。
5. 对文本进行分块。
6. 使用 embedding 模型生成向量。
7. 向量写入数据库并同步进入 FAISS 索引。

### 问答流程

1. 用户输入问题。
2. 系统将问题转为向量。
3. 在 FAISS 中检索 top-k 相关分块。
4. 拼接上下文。
5. 调用大模型生成最终回答。
6. 返回答案和参考来源。

### 图文联合问答流程

1. 用户输入问题并附加图片。
2. 系统仍然先从知识库检索相关文本。
3. 再把“问题 + 检索结果 + 图片”一起传给多模态模型。
4. 最终生成兼顾图片信息和知识库内容的回答。

## 6. 项目亮点

你答辩时可以重点强调这几个点：

1. 从纯文本 RAG 扩展到了多模态输入。
2. 检索层不再是简单遍历，而是接入了 FAISS。
3. 系统支持图片入库，不只是上传图片后直接问答。
4. 做了工程化改进，包括配置安全化、文件名清洗、失败清理和前端安全渲染。

## 7. 创新点与价值

本项目的创新点不在于自研模型，而在于针对工业场景进行了工程整合和能力扩展：

1. 将工业图片转为可检索语义摘要，实现图片知识入库。
2. 将文本检索与图文问答统一到一个系统中。
3. 用较轻量的 Flask + FAISS 架构实现可运行、可展示、可扩展的工业知识问答平台。

它的实际价值主要体现在：

- 提高工业资料查询效率。
- 降低人工查阅手册和设备说明书的成本。
- 为工业知识助手、故障辅助分析等方向提供一个原型基础。

## 8. 已完成内容

当前已经完成：

1. 基础前后端界面与接口联调。
2. 文档和图片上传入库。
3. 文本和图片的知识抽取与向量索引。
4. 文本问答与图文问答。
5. 历史记录、知识库统计和参考来源展示。

## 9. 不足与后续优化

这个部分老师一般会问，建议你主动说：

1. 当前图片理解依赖云端多模态模型，离线能力有限。
2. OCR、表格结构化、图像区域定位还可以继续增强。
3. 检索目前主要是向量检索，还可以增加关键词检索和重排序模型。
4. 目前缺少权限管理、用户体系和更完整的日志监控。
5. 如果面向企业部署，还需要进一步优化并发、缓存和索引更新机制。
